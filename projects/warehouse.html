Warehouse Management System

An ASP.NET Core 8 internal tool for tracking inventory across multiple warehouse locations, built around atomic stock movements, role-based access, and a clean-architecture layout that keeps the domain free of framework dependencies.

Overview

The system models a real warehouse: products live in locations, stock at each location is tracked as an inventory item, and every change is recorded as a stock movement (Receive, Ship, or Adjustment). The dashboard surfaces totals, low-stock alerts, recent activity, and a 7-day in/out trend.

I deliberately structured it as a clean-architecture solution rather than a single Razor app, keeping the domain pure and the data layer swappable made it much easier to write fast, isolated tests for the stock-movement logic.

Architecture

The solution is split into four projects, each with a single responsibility:

  • WarehouseApp.Core - pure C# domain. Entities and service interfaces only. No EF, no ASP.NET, no dependencies.
  • WarehouseApp.Infrastructure - EF Core 8 + MySQL (Pomelo provider), the ApplicationDbContext, migrations, seed data, and concrete service implementations.
  • WarehouseApp.Web - ASP.NET Core MVC with Razor Views, Identity for auth, and controllers for every resource.
  • WarehouseApp.Tests - xUnit suite running against EF Core's InMemory provider for fast, isolated tests.

Domain Model

  • Product - SKU, name, description, unit price, reorder level
  • Location - code + description (e.g. BIN-A1, DOCK-3)
  • InventoryItem - quantity of a product at a specific location
  • StockMovement - append-only record of every Receive / Ship / Adjustment with quantity, reference number, notes, performing user, and timestamp

Key Features

  • Products & Locations CRUD - full create / edit / delete with validation
  • Inventory view - stock-on-hand by product and location
  • Receive flow - pull stock in; auto-creates the inventory row if it doesn't exist yet
  • Ship flow - pull stock out, with oversell prevention (rejects if requested qty exceeds available)
  • Movement history - every change is logged with user, timestamp, and reference number
  • Dashboard - totals (products, locations, units), low-stock alerts based on each product's reorder level, recent movements, and a 7-day in/out chart
  • Identity-based auth - ASP.NET Core Identity gates all admin operations and stamps every movement with the performing user ID

Technical Highlights

  • Atomic stock movements - each Receive/Ship runs inside a database transaction (BeginTransactionAsync → CommitAsync) so the movement record and inventory adjustment commit or roll back together
  • Oversell prevention - Ship checks current available qty against the requested amount before writing anything
  • Tested service layer - 7 xUnit tests covering happy paths and edge cases (zero quantity, unknown product, no inventory, oversell)
  • Result-type responses - services return a StockOperationResult with Success + optional error message instead of throwing on validation failures
  • EF Core migrations - schema evolution is version-controlled and reproducible

Tech Stack

ASP.NET Core 8 C# / .NET 8 Entity Framework Core MySQL (Pomelo) ASP.NET Identity Razor Views Bootstrap xUnit
