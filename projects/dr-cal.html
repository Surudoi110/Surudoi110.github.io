Dr. Cal — Calorie Tracker

A student-friendly calorie tracker. Mobile-first React 19 PWA on the front, Laravel 12 + Sanctum on the back — with token auth, meal & exercise logging, a live daily summary, and 14 days of history.

full-stack mobile-first Laravel 12 + React 19 2026
[ screenshot.png — drop your dashboard or history screenshot here ]

Overview

Dr. Cal is a complete two-tier app, designed for the small-screen "phone in hand" use case. The frontend is a React 19 SPA built with Vite that talks to a Laravel 12 API over JSON. Every protected route waits for a Sanctum bearer token — register or log in, and the token is cached in localStorage and attached to every subsequent request through an axios interceptor.

The dashboard isn't static. It pulls from /api/today, which assembles meals grouped by type, exercise totals, and a smart tip on every request — so the screen reflects whatever you logged in the last few seconds.

Architecture

       ├─ AuthContext  →  localStorage (token + user)
       ├─ axios client →  Bearer interceptor + 401 auto-redirect

       ▼ JSON over HTTP

[SQLite] users · meals · exercises · personal_access_tokens

Database Schema

  • users — id, name, email (unique), password (hashed), timestamps
  • meals — id, user_id (FK cascade), name, calories, meal_type (Breakfast / Lunch / Dinner / Snack), consumed_at, timestamps
  • exercises — id, user_id (FK cascade), name, calories_burned, performed_at, timestamps

Key Features

  • Auth flow — register, login, logout, with auto-login after sign-up so users land on the dashboard immediately
  • Protected routes — a ProtectedRoute wrapper redirects unauthenticated users back to /login and remembers where they were trying to go
  • Dashboard — progress ring rendered with conic-gradient showing % of daily target, intake / burn / net stat strip, meal cards by type, and a smart tip that adjusts to where the user is in their day
  • Add Food — meal-type select pre-fills from a ?type= query param so the dashboard's "+" buttons feel native
  • Log Exercise — quick-pick chips for common activities (Walking / Running / Cycling / Yoga / Swimming) with estimated kcal, plus a free-form form
  • 14-day History — per-day net with a progress bar that turns red when over target, plus intake↑ / burn↓ breakdown
  • Pro page — feature comparison table, mock upgrade button

Technical Highlights

  • Sanctum personal access tokens for stateless bearer auth — no CSRF dance, easy to swap to native mobile clients later
  • Ownership-checked destroy endpoints — both MealController::destroy and ExerciseController::destroy verify user_id before deleting (the original prototype had a bug that let any authenticated user delete any meal)
  • Server-computed daily summary — instead of doing math in the browser, SummaryController::today returns the full payload (totals, groupings, remaining, tip) so the frontend just renders. Easier to test, easier to swap clients.
  • Meal-type as enum — validated server-side via Rule::in(Meal::TYPES) so the four-section dashboard layout can never be broken by bad data
  • Smart tip generator — the backend looks at the user's % of target and returns a contextual nudge ("solid pace", "almost at your target", "consider a short walk")
  • CORS locked to dev origin — localhost:5173 only, configurable via FRONTEND_URL env var
  • PWA-ready — Vite project structure is set up for vite-plugin-pwa registration; the SW hook is left as a one-liner to enable
